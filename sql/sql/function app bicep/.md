Integration Guide: Adding Function Apps to Existing Pipeline
This guide shows you how to add the 4 Function App deployments to your existing main.bicep and use your current pipeline.

📋 Prerequisites
Your existing structure should have:

✅ main.bicep in Infrastructure/NDR/Bicep/Bridge/environments/dev/
✅ main.parameters.json in the same folder
✅ Existing pipeline that deploys your infrastructure
✅ modules/ folder for Bicep modules
🔧 Step 1: Add the Function App Module
Location: Infrastructure/NDR/Bicep/Bridge/modules/Compute/functionApp.bicep

Create the Compute folder if it doesn't exist:
   Infrastructure/NDR/Bicep/Bridge/modules/Compute/
Copy the functionApp.bicep module file into this folder
📝 Step 2: Update Your Existing main.bicep
Open your existing main.bicep file and add the following sections:

A. Add Parameters (at the top with your other parameters)
bicep
// Function App Parameters - ADD THESE
@description('.NET version for Function Apps')
param dotnetVersion string = 'v10.0'

@description('Minimum elastic instance count')
param minimumElasticInstanceCount int = 1

@description('Integration subnet name for VNet integration')
param integrationSubnetName string

@description('Private endpoint subnet name')
param privateEndpointSubnetName string
B. Add Module Calls (after your existing modules)
Add these 4 module calls after your existing module declarations (like storageAccount, appInsights, etc.):

bicep
// Function App 1: CDB to Bridge ETL
module cdbToBridgeFunction '../modules/Compute/functionApp.bicep' = {
  name: 'deploy-cdb-bridge-function'
  params: {
    location: location
    functionAppName: 'vos-iaz-func-${environment}-bridge-cdb-bridge-etl'
    functionAppDotnetFrameworkVersion: dotnetVersion
    functionAppVirtualNetworkResourceGroupName: vnetResourceGroupName
    functionAppVirtualNetworkName: vnetName
    functionAppSubnetName: integrationSubnetName
    functionAppPrivateEndpointSubnetName: privateEndpointSubnetName
    functionAppUserAssignedManagedIdentityClientId: identities[0].clientId
    functionAppUserAssignedIdentityId: identities[0].id
    functionAppPlanId: appServicePlanId
    functionAppMinimumElasticInstanceCount: minimumElasticInstanceCount
    applicationInsightsConnectionString: appInsights.outputs.connectionString
    tags: union(tags, {
      Purpose: 'CDB to Bridge ETL'
      FunctionType: 'DataTransport'
    })
  }
  dependsOn: [
    appInsights
  ]
}

📄 Step 3: Update Your main.parameters.json
Open your existing main.parameters.json and add these parameters:

json
{
  "$schema": "https://schema.management.azure.com/schemas/2019-04-01/deploymentParameters.json#",
  "contentVersion": "1.0.0.0",
  "parameters": {
    
    // ... your existing parameters ...
    
    "dotnetVersion": {
      "value": "v10.0"
    },
    "minimumElasticInstanceCount": {
      "value": 1
    },
    "integrationSubnetName": {
      "value": "snet-integration"
    },
    "privateEndpointSubnetName": {
      "value": "snet-private-endpoints"
    }
  }
}

🔍 Step 4: Validate Your Changes
Before committing, validate the Bicep template:

bash
# Navigate to your template directory
cd Infrastructure/NDR/Bicep/Bridge/environments/dev

# Validate
az deployment group validate \
  --resource-group C190VOArgpVOSDEVBRIDGE \
  --template-file main.bicep \
  --parameters main.parameters.json

🆘 Troubleshooting
Issue: Module not found
Error: Unable to find module '../modules/Compute/functionApp.bicep'

Solution: Check the relative path from your main.bicep to the module file.

Issue: Parameter not provided
Error: The parameter 'integrationSubnetName' was not provided

Solution: Ensure you added all new parameters to main.parameters.json

Issue: Identity not found
Error: Identity with index [X] does not exist

Solution: Ensure your identities array has 4 items in the parameters file

Issue: Subnet delegation error
Error: Subnet must be delegated to Microsoft.Web/serverFarms

Solution:

bash
az network vnet subnet update \
  --resource-group C190VOArgpVOSDEVBRIDGE \
  --vnet-name YOUR_VNET_NAME \
  --name snet-integration \
  --delegations Microsoft.Web/serverFarms
